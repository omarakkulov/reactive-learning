# Deep Dive: ОС, потоки, сокеты и ожидание I/O

Этот файл - инженерное углубление к первому блоку. Его не обязательно давать всей группе сразу.
Основной README должен быть понятен без этих деталей, а здесь можно спокойно разобрать, что происходит
ниже уровня Spring.

## Процесс и поток

Процесс - это запущенная программа со своим адресным пространством.

Поток - это единица выполнения внутри процесса. У потока есть:

- instruction pointer: где сейчас выполняется код;
- stack: локальные вызовы и переменные;
- registers: текущее состояние CPU для выполнения;
- состояние в планировщике ОС.

В Java поток `java.lang.Thread` обычно соответствует нативному потоку ОС. Это удобно, но не бесплатно.
Каждый поток требует памяти под stack и участвует в планировании ОС.

## Context switching

CPU core в конкретный момент выполняет конкретный поток. Если runnable-потоков больше, чем cores,
операционная система переключает их.

Context switch - это сохранение состояния одного потока и восстановление состояния другого.

Это нормально и необходимо, но у этого есть цена:

- CPU тратит время не на бизнес-код;
- кэши процессора могут становиться менее полезными;
- при большом количестве потоков растет накладная стоимость планирования.

Поэтому модель "создадим очень много потоков, и все будет хорошо" имеет предел.

## Blocking I/O с точки зрения ОС

Когда поток делает blocking I/O, он часто переходит в состояние ожидания. Например, поток ждет:

- данные из socket;
- ответ от диска;
- завершение sleep/timer;
- освобождение lock.

Пока поток ждет, CPU может выполнять другие потоки. Но сам ожидающий поток остается ресурсом:
у него есть stack, состояние, участие в планировании и жизненный цикл.

В thread-per-request модели это выглядит так:

```text
1000 одновременных запросов -> до 1000 потоков, которые могут стоять в ожидании I/O
```

Если каждый запрос большую часть времени ждет БД или внешний HTTP-сервис, много потоков будут заняты
не вычислениями, а ожиданием.

## Socket

Socket - это абстракция ОС для сетевого соединения.

Для TCP-соединения можно упрощенно думать так:

```text
client socket <---- network ----> server socket
```

Сервер принимает соединения, читает байты, пишет байты обратно. HTTP - это уже протокол поверх этих
байтов: request line, headers, body, response.

## Blocking socket read

В blocking-модели чтение может выглядеть концептуально так:

```text
read(socket)
```

Если данных пока нет, поток ждет. Для простого кода это удобно: мы пишем последовательную логику.
Но при большом количестве соединений такая модель требует много потоков.

## Non-blocking socket и readiness

В non-blocking-модели socket можно перевести в режим, где операция чтения не обязана ждать.
Если данных нет, операция быстро возвращает "сейчас читать нечего".

Тогда появляется другая задача: как узнать, какие из тысяч соединений готовы к чтению или записи?

Для этого ОС предоставляет механизмы readiness notification:

- `select`;
- `poll`;
- `epoll` на Linux;
- `kqueue` на macOS/BSD;
- IOCP на Windows.

Идея:

```text
1. Зарегистрировать интерес к событиям на множестве socket.
2. Спросить ОС: какие socket сейчас готовы?
3. Быстро обработать готовые события.
4. Вернуться к ожиданию следующих событий.
```

## Selector

В Java NIO есть `Selector`. Он позволяет одному потоку следить за несколькими channels.

Упрощенная модель:

```text
while (true) {
    selector.select();
    for (readyKey : selector.selectedKeys()) {
        handle(readyKey);
    }
}
```

Это уже похоже на event loop: поток крутится вокруг событий, а не закрепляется за одним запросом.

## Event loop

Event loop - это цикл обработки событий.

В Reactor Netty event loop поток обслуживает множество channels. Он должен:

- принимать сетевые события;
- читать и писать байты;
- прогонять данные через pipeline;
- передавать события выше;
- быстро возвращаться к следующему событию.

Если внутри event loop сделать blocking operation, этот поток перестает обслуживать остальные channels
на время блокировки.

Именно поэтому `Thread.sleep`, JDBC, blocking HTTP client или тяжелая CPU-работа внутри event loop -
опасные операции.

## Почему `Thread.sleep` полезен как учебный пример

`Thread.sleep` не является сетевым I/O, но он хорошо показывает суть blocking:

```text
Поток есть, но полезную работу не делает.
```

В реальном приложении вместо `sleep` часто оказывается:

- blocking JDBC driver;
- `RestTemplate`;
- синхронный SDK внешнего сервиса;
- чтение файла blocking API;
- случайный `.block()` внутри reactive chain.

Механика для event loop похожая: поток занят ожиданием и не может быстро обслуживать другие события.

## Важная граница понимания

Non-blocking модель не отменяет ожидание внешнего мира. База данных, сеть и другой сервис все еще
имеют свою latency.

Non-blocking модель меняет вопрос:

```text
Что делает поток, пока мы ждем?
```

В blocking-модели поток обычно стоит. В non-blocking-модели поток может обслуживать другие события.
