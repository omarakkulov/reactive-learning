# День 1. Blocking vs Non-Blocking

Считаю, главная задача первого урока - не научить людей орудовать операторами `map`, `flatMap` и внутренностями `Mono`, `Flux`. До этого мы
дойдем позже. Как по мне, сначала нужно собрать внутреннюю модель: понять проблемы, возникшие в блокирующем подходе, понять фундаментальные
сетевые отличия этих подходов, и заодно вспомнить, что такое процесс, поток, ожидание I/O, почему thread-per-request(один поток на один
запрос) имеет предел и зачем вообще появилась реактивная модель.

## 0. Что человек должен понять после урока

Перед началом каждого урока я буду говорить о том, чего примерно ожидаю от каждого слушателя в конце этого урока.
А именно, я очень постараюсь, чтобы каждый слушатель смог суметь объяснить несколько важнейших вещей в рамках разработки простыми словами:

- что происходит, когда мы запускаем Java-приложение;
- чем процесс отличается от потока;
- почему поток - это не бесплатный ресурс;
- что делает scheduler операционной системы;
- почему blocking I/O занимает поток ожиданием;
- почему async и non-blocking - это разные вещи;
- что такое socket, selector и event loop на уровне идеи, а дальше же мы вернемся к нему гораздо глубже для понимания всех реактивных
  процессов;
- почему WebFlux нельзя писать как обычный blocking MVC-код, просто завернув результат в `Mono`;
- какую проблему решают Reactive Streams и реактивное программирование.

Реактивность не ускоряет внешний мир, а меняет концепцию того, как приложение использует потоки во время ожидания данных от различных I/O
операций. То есть, если база данных отвечает 300 ms, реактивность не превратит эти 300 ms в 3 ms. Но она может помочь приложению не держать
поток заблокированным все это время.

## 1. Backend-сценарий

Представим, у нас есть сервис `MyApp`, в нем доступен endpoint:

```text
GET /clients/123/profile
```

Чтобы вернуть ответ, сервису нужно:

1. сходить в БД за клиентом;
2. сходить в `account-service` за счетами;
3. сходить в `limit-service` за лимитами;
4. сходить в `transaction-service` за последними операциями;
5. собрать JSON-ответ.

В blocking-модели это часто выглядит так:

```text
HTTP request
   ↓
worker thread
   ↓
Controller
   ↓
Service
   ↓
DB call             ← поток ждет
   ↓
HTTP account call   ← поток ждет
   ↓
HTTP limits call    ← поток ждет
   ↓
Response
```

Главный вопрос текущего пункта:

```text
Что значит "поток ждет" с точки зрения операционной системы?
```

Чтобы ответить на этот вопрос, нам нужна минимальная модель ОС: процесс, поток, scheduler, состояния потоков и I/O.

## 2. Process / Thread / CPU / RAM / Scheduler

![cpu_process.png](../../../../../../resources/static/lesson01/cpu_process.png)

Предположим, мы разработали простое Java-приложение и упаковали его в Jar-файл.
Этот Jar, соответственно, будет храниться где-то на диске и наша программа - это не что иное, как некий набор инструкций, которые
необходимо выполнить.

Когда мы пишем команду

````
java -jar MyApp.jar
````

То MyApp будет загружен в RAM, как Process. Или, по другому, ОС создает процесс, подготавливает для него адресное пространство, загружает
нужные части программы и библиотек в память, выделяет ресурсы и запускает главный поток. И сейчас будет несколько не слишком понятных
определений, которые я поясню:

**Процесс** - это абстракция, существующая на программном уровне - уровне операционной системы.
Создание процесса - тяжеловесная процедура на основе системных вызовов.
Процесс в текущем контексте можно трактовать, как запущенное приложение со своей изолированной, выделенной конкретно под нее, областью
памяти.

**Поток** - это единица выполнения внутри процесса. То есть, поток - часть процесса.

Итак, процесс — это экземпляр компьютерной программы. Он включает в себя наш код, другие ресурсы, выделенные ОС: сокеты, память и тп

**У потока есть:**

- **instruction pointer** (счетчик, или наш «палец» на строчке кода) - Это «навигатор или закладка», который указывает процессору, какую
  строчку кода выполнять следующей.

Когда поток в работе (CPU): Это физический регистр внутри ядра процессора. Значение в нём меняется с бешеной
скоростью (миллиарды раз в секунду), пока поток выполняет строки кода.

Когда поток в покое (RAM): Это число, а вернее адрес в памяти, или же, можно сказать, строчка кода, сохранённая в специальную структуру
данных в оперативной памяти(TCB - Thread Control Block). ОС записывает его туда только в момент остановки потока.

Еще раз, как работает и где живет: Когда поток работает — это физическая ячейка внутри CPU. Когда ОС переключает процессор на другой поток,
она записывает значение этой физической ячейки внутри CPU в память (в контекст потока), чтобы потом начать ровно с того же места.
Когда поток уходит с процессора, ОС делает «мгновенное фото - snapshot» всех регистров и сохраняет этот контекст в RAM. Когда поток
возвращается на процессор, ОС «заливает» эти данные обратно в регистры.

Крайняя точка: Если instruction pointer укажет на адрес, где нет кода (или это будет код другого процесса), ОС немедленно прибьет
приложение с ошибкой Segmentation Fault.

- **stack**: локальные вызовы и переменные;

Когда поток в работе (CPU + RAM): Стек всегда находится в оперативной памяти (RAM). Однако те части стека, с которыми процессор работает
прямо сейчас, автоматически подтягиваются в кэш процессора (L1/L2), чтобы всё летало.

Когда поток в покое (RAM): Стек остаётся в оперативной памяти в неизменном виде. ОС просто запоминает адрес начала этого стека в том же
TCB - в той самой специальной структуре данных оперативной памяти

Крайняя точка: Если методы вызывают друг друга бесконечно, выделенный под стек мегабайт в RAM заканчивается, и поток
погибает - StackOverflowError

- **registers**: (Регистры общего назначения) — Это «руки» процессора, в которых он держит данные во время вычислений.

*По аналогии с instruction pointer и stack*

Когда поток в работе (CPU): Это набор ячеек внутри процессора. Если ты складываешь a + b, то значения a и b физически лежат в этих
транзисторных ячейках. Они — самое быстрое, что есть в мире компьютеров.

Когда поток в покое (RAM): Это Snapshot (снимок). Поскольку регистров мало и они одни на всех, при паузе ОС копирует все значения из
«железных» регистров в TCB в оперативную память.

**Процесс будет содержать хотя бы один поток. Очевидно, их может быть гораздо больше.**

### Про выделение памяти при запуске Java-процесса

Здесь важно сделать техническую поправку.

Когда мы запускаем:

```text
java -jar MyApp.jar
```

операционная система не запускает сам `MyApp.jar` как обычный нативный исполняемый файл.

ОС запускает программу `java`, то есть Java launcher/JVM-процесс. А уже JVM внутри этого процесса:

- открывает JAR-файл;
- читает manifest и классы;
- загружает нужные классы через classloader;
- подготавливает runtime-структуры;
- запускает главный Java-поток;
- вызывает `public static void main(...)`.

То есть схема такая:

```text
OS создает процесс для java
        ↓
JVM стартует внутри этого процесса
        ↓
JVM читает MyApp.jar
        ↓
JVM загружает классы
        ↓
JVM запускает main Java thread
```

Поэтому фразу «ОС загружает JAR как процесс» лучше понимать как упрощение. Точнее:

```text
ОС создает JVM-процесс,
а JVM уже загружает Java-приложение из JAR.
```

Для Java-приложения полезнее думать не в терминах `Code Segment / Data Segment / ReadOnly Segment` самого JAR-файла, а в терминах памяти
JVM-процесса.

У JVM-процесса есть виртуальное адресное пространство, внутри которого живут разные области:

```text
JVM Process
┌─────────────────────────────────────────────────────────────┐
│ Virtual Address Space                                       │
│                                                             │
│  Java Heap                  — объекты Java                  │
│  Thread Stacks              — стек каждого platform thread  │
│  Metaspace                  — metadata классов              │
│  Code Cache                 — JIT-скомпилированный код      │
│  Native Memory              — память JVM/native-библиотек   │
│  Loaded Libraries           — libc, JVM libs и т.п.         │
│  File Descriptors/Sockets   — открытые файлы и соединения   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

Память не обязательно физически выделяется вся сразу в RAM. Современная ОС работает через виртуальную память: она создает адресное
пространство и отображения, а реальные физические страницы могут подтягиваться лениво, по мере доступа.

Если мы создаем новый platform thread, JVM и ОС должны выделить ресурсы для этого потока, включая память под stack. Размер stack зависит от
JVM, ОС и настроек, например от `-Xss`. Практически это часто сотни килобайт или мегабайты на поток, поэтому десятки тысяч platform threads
становятся дорогими.

Если мы создаем объект:

```java
new Object()
```

объект обычно размещается в Java Heap. JVM сначала пытается выделить память внутри уже доступной heap-области. Если места недостаточно, JVM
может запустить GC, расширить heap в рамках `-Xmx` или завершиться с `OutOfMemoryError`, если память получить уже нельзя.

Для нашего урока не нужно глубоко уходить в устройство heap, metaspace, page faults и native memory. Нам нужна главная мысль:

```text
Процесс владеет ресурсами.
Потоки исполняют код внутри процесса.
Каждый platform thread имеет свою стоимость.
```

---
Операционная система выделяет память под процесс и эта память совместно используется потоками, а значит:

#### Процесс - единица ресурсов, а поток - единица выполнения.

**Состояния потоков:** Для нашего урока достаточно трех

- **Выполняемый (Executing)** - поток, который выполняется в текущий момент на процессоре
- **Заблокированный (Waiting)** - работа потока заблокирована в ожидании блокирующей операции
- **Готовый (Runnable или Ready)** - поток ждет получения кванта времени и готов выполнять назначенные ему инструкции. Планировщик выбирает
  следующий поток для выполнения только из готовых потоков.

Процесс распределения ресурсов происходит в несколько этапов:

В ОС есть нечто, называемое **Планировщиком(Scheduler)** - он назначает поток на процессор, управляя очередями задач и переключая контекст
выполнения, в то время как исполнение конкретных инструкций контролируется самим процессором на аппаратном уровне.
А если проще, то он анализирует список потоков в состоянии «Готов» (Ready) и
выбирает один из потоков для выполнения на основе определенных правил.

**1) Выбор потока (Алгоритмы планирования)**

- **Приоритетность**: Потоки с более высоким приоритетом получают доступ к процессору раньше остальных.
- **Квантование времени**: Каждому потоку выделяется фиксированный отрезок времени (квант). Когда он истекает, ОС принудительно прерывает
  поток, чтобы дать поработать другим.
- **Affinity (Сродство)**: ОС старается закрепить поток за конкретным ядром процессора, чтобы эффективнее использовать кэш-память этого
  ядра.

**2) Переключение контекста (Context Switch)**
Чтобы запустить выбранный поток, планировщик выполняет следующие действия:

- **Сохранение**: Состояние текущего потока (значения регистров, указатель команд) записывается в специальную структуру памяти в ОС - в RAM
- **Загрузка**: Состояние нового потока восстанавливается в регистры процессора.
- **Переход**: Указатель команд - устанавливается на следующую инструкцию нового потока, и процессор начинает выполнение

Здесь важно уточнение.

Когда дальше я говорю `Java thread`, я в первую очередь имею в виду **platform thread**.

Классический platform thread в Java — это тонкая оболочка вокруг потока ОС: один Java platform thread обычно соответствует одному native OS
thread. Именно про такую модель мы говорим, когда обсуждаем Spring MVC, servlet worker threads, Netty event-loop threads и стоимость
большого количества потоков.

Начиная с Java 21, есть еще **virtual threads**. Virtual thread — это тоже `java.lang.Thread`, но он не привязан 1:1 к отдельному OS thread на
все время своей жизни. Это отдельная большая тема. Для этого урока достаточно зафиксировать:

```text
Platform thread ≈ OS thread
Virtual thread  ≠ отдельный OS thread на весь lifetime
```

В текущем уроке мы обсуждаем именно platform threads, потому что Netty EventLoop — это реальные долгоживущие platform/OS threads.

**Так в чем проблема подобного подхода? Мы дошли до самого интересного**

- CPU дорогой, причем речь тут о финансовой составляющей. И попытка развернуть что-либо на каких-то серверах это подтвердит,
  учитывая цену на аренду машин. И нам следует использовать этот процессор как можно рациональнее и больше, дабы он впустую не
  простаивал и не тратил деньги.
- Часто в архитектуре микросервисов у нас слишком много сетевых вызовов, и сетевые вызовы медленны. В рамках такого вызова, поток,
  отправивший запрос во внешний сервис, будет заблокирован до тех пор, пока не придет ответ, а значит, что процессор впустую простаивает.
  И бывает такое, что если сервис работает в рамках блокирующего подхода, и сетевых вызовов ожидается много, то и потоков нужно очень
  много, а каждый поток - занимает память.
- А значит, сам поток - тяжеловесный ресурс, так как потребляет порядка 1 МБ памяти при создании.
- Поэтому нам нужен механизм, который сделает эти сетевые вызовы более эффективными, не тратя впустую системные ресурсы.

## 3. IO Models: Inbound / Outbound

Поговорим о различных моделях ввода/вывода:

![io_models.png](../../../../../../resources/static/lesson01/io_models.png)

1) **sync + blocking** — Очень простой и понятный синхронный блокирующий обмен данными, с которым мы часто сталкивались.

- **ПРИМЕР**: Я звоню в парикмахерскую по предоставленному номеру. Я могу ждать ответа сколь угодно долгое время и не факт, что
  получу его. До тех пор, пока не придет ответ, все это время мне пришлось ждать и сидеть без дела. Я не могу сделать ничего другого, потому
  что не знаю, когда кто-нибудь будет доступен на другом конце провода.

2) **async** - На этот раз вместо того, чтобы позвонить в парикмахерскую, я спросил друга: «Эй, приятель, помоги кое с чем, позвони,
   пожалуйста, в парикмахерскую, и запиши меня на завтра». В этом случае я делегировал задачу второму потоку, который будет ждать, пока
   парикмахерская не ответит. Как видно, я не заблокирован, поскольку делегировал задачу, однако, мой друг в этом сценарии будет
   заблокирован в рамках своего вызова, а значит, мы наблюдаем аналогичный сценарий, что какой-то поток простаивает.

3) **non-blocking**: Предположим, что я опять звоню в парикмахерскую. Автоматизированная система голосовых сообщений на стороне
   парикмахерской сообщает, что все мастера в данный момент заняты, оставьте свой номер телефона и мы вам перезвоним, когда кто-то
   освободится. Я даю им свой номер телефона и могу заниматься своими делами и дальше, но через некоторое время мой телефон зазвонит и
   мне сообщат о том, какой мастер освободился. В этом случае я, как поток, отправил запрос и не заблокировался. В этом случае ОС
   уведомит поток о том, что ответ от внешнего сервиса был получен. Этот сценарий в рамках курса мы обязательно еще обсудим подробнее

4) **non-blocking + async**: это комбинация **async и non-blocking** подхода одновременно. Предположим, я опять звоню в парикмахерскую,
   мне ответят "Оставьте ваши контакты для связи, мы вам перезвоним". В этом случае, вместо того, чтобы давать свой номер, я даю номер
   друга, чтобы перезвонили именно ему, а значит, я не заблокирован, и мой друг не заблокирован. Аналогично, поток в этом случае шлет
   запрос во внешнюю систему, занимается своими делами дальше, и уже ОС УВЕДОМИТ другой поток(моего друга) для обработки ответа

И если рассматривать сложность разработки по шкале от 1 до 4, то в таком же порядке она и будет:

1) **sync + blocking**
2) **async**
3) **non-blocking**
4) **non-blocking + async**

Реактивное программирование - это модель программирования, упрощающая неблокирующую асинхронную связь.

## 4. Типы коммуникаций

Мы писали код так, что отправляли запрос, и ждали ответ, и так писали много-много лет. Это простое требование, в котором можно обойтись и
без реактивности, подумав над архитектурой общения.
Можно ли использовать Virtual Threads в таком сценарии? - можно использовать, более того, нужно использовать, если позволяет система.

Однако, мы можем решить множество других проблем и сейчас я объясню, каких именно.
Реактивное программирование открывает двери для трех дополнительных моделей общения на выбор:

![communication_patterns.png](../../../../../../resources/static/lesson01/communication_patterns.png)

1) **request -> response**: Все понятно
2) **request -> streaming response**: Мы отправляем запрос и получаем множественный ответ. Предположим, мы хотим заказать пиццу,
   отправляя запрос, и в ответ нам могут слать не просто некий http-ответ, а потоковый multiple-response ответ, например получаем
   потоковый, динамически заполняющийся в режиме реального времени ответ. Например, смотрим в смартфон и видим, что заказ перешел с одного
   статуса в другой, видим движение курьера на карте и тп. То есть, получаем Streaming-response
3) **streaming request -> response**: В некоторых случаях мы и сами можем слать streaming-request в удаленный сервер. Например, у нас
   есть условные умные часы, которые периодически могут слать данные о частоте пульса на удаленный сервер и тп, и это можно реализовать,
   используя возможности реактивного программирования. Либо, к примеру, мы работаем с Google документом на удаленном сервере, изменяя
   его время от времени, и изменения сохраняются на сервере. В этом случае надо еще понимать, что мы не шлем несколько http-запросов в
   цикле, а открываем одно соединение, через которое отправляем несколько потоковых сообщений на удаленный сервер.
4) **Bidirectional-streaming**: это комбинация 2 и 3 подхода(двунаправленный стриминг данных). Приложения могут продолжать обмениваться
   данными динамически(аналог диалога между людьми)

Все типы указанных коммуникаций возможны в реализации в реактивной системе

## 5. Почему thread-per-request имеет предел

В классической servlet-модели, например в Spring MVC, удобно думать так:

```text
request -> worker thread -> controller -> service -> repository/client -> response
```

Один входящий запрос занимает worker thread. Этот поток проходит через controller, service, repository или HTTP client, и в конце
возвращает response.

Пока нагрузка небольшая, это отличная модель: она простая, последовательная, понятная, хорошо ложится на обычный Java-код.

Проблема начинается там, где внутри обработки запроса много ожидания:

- запрос в базу данных;
- HTTP-вызов в другой микросервис;
- обращение к брокеру;
- чтение файла;

Если все эти операции блокирующие, поток ждет. Он не считает. Он не делает полезную CPU-работу. Он просто удерживает ресурс, потому что
логика написана последовательно: "сходи туда, дождись ответа, потом продолжи".

Еще раз аккуратная формулировка:

```text
CPU не обязан простаивать целиком.
Но request-thread занят ожиданием и не может взять другой запрос.
```

Если таких запросов 1000, то в blocking thread-per-request модели нам может понадобиться очень много потоков, которые значительную часть
времени будут просто ждать I/O.

![microservices-communication.png](../../../../../../resources/static/lesson01/microservices-communication.png)

Технологические гиганты, такие как Twitter, Netflix и тп, например, понимают, что приложения сейчас становятся все более сложными. Сейчас
люди используют смартфоны, умные часы и т. д. И почти все они подключены к Интернету, а значит, например, клиентов интересуют некие
механизмы получения различных уведомлений в режиме реального времени и тп.

То есть, типичное общение в стиле запроса и ответа не очень помогает, потому что мы не можем продолжать отправлять запросы в цикле к
удаленному серверу по типу: «Эй, сервер, у тебя есть обновления?» и так 20 раз. Не очень разумным будет подход, когда мы периодически
продолжаем отправлять запрос каждую секунду, так как это неэффективно. Незачем просить ресурс тогда, когда его нет, либо
он не готов. Вместо этого, в идеале, сервер сам должен уведомлять клиента в случае обновлений, например, в виде потокового ответа(streaming
response). Поэтому, если модель общения между приложениями сложная, то типичный request-response стиль не поможет и нам нужна иная
программная модель.

## 6. Reactive-streams

![reactive-streams.png](../../../../../../resources/static/lesson01/reactive-streams.png)

И вот, примерно в 2014 году, была создана новая спецификация, называемая **Reactive Streams**. Она определяет стандарт асинхронной обработки
потоков данных с неблокирующим противодавлением.

Например, беря в аналогию объектно-ориентированное программирование с его абстракциями, инкапсуляцией, наследованиями и полиморфизмами, то,
аналогично, для реактивного программирования ключевыми концептуальными определениями будут выступать: **asynchronous**, **non-blocking**,
**backpressure**, **stream processing**, где "stream processing" - это не одно сообщение, как мы говорили сегодня ранее, это поток
сообщений.

Путем обсуждения проблем и решений, это создало необходимость в другой модели программирования - реактивном программировании

## 7. Что такое реактивное программирование?

Как мы поняли по определению, это новая парадигма программирования, предназначенная для обработки потока сообщений неблокирующим
асинхронным способом управления с обратным давлением и она обещает упростить неблокирующее асинхронное взаимодействие

Также важно отметить, что парадигма основана на шаблоне проектирования Наблюдатель(Observer)

Реактивное программирование лучше всего себя показывает при множественных I/O вызовах и дополняет объектно-ориентированное программирование,
предоставляющее мощные инструменты для обработки этого асинхронного неблокирующего процесса

## 8. CPU-bound и I/O-bound работа

Чтобы понять, где реактивность полезна, нужно различать два типа работы.

#### CPU-bound работа:

- посчитать хэш;
- обработать большой JSON;
- выполнить сложный алгоритм;
- сделать тяжелую криптографию;

Здесь узкое место - CPU. Если CPU занят вычислениями, non-blocking сам по себе не решит проблему. Нужны parallelism, отдельные scheduler'ы,
ограничение concurrency и понимание границ выполнения.

#### I/O-bound работа:

- сходить в базу данных;
- отправить HTTP-запрос;
- получить данные из Redis;
- дождаться Kafka;
- прочитать файл;
- дождаться ответа от внешнего сервиса.

Здесь узкое место - внешний мир. Сеть, диск, база данных и другой сервис намного медленнее CPU. Backend-приложение часто большую часть
времени не считает, а ждет.

Именно здесь non-blocking модель становится особенно интересной:

Интуитивно можно прийти к вопросу

```text
Если мы все равно ждем внешний мир, можно ли не удерживать поток во время ожидания?
```


## 9. Фундамент реактивного программирования: Netty, EventLoop и очереди

До этого момента мы обсуждали проблему блокирующего подхода:

```text
много concurrent I/O-операций
    ↓
много ожидания
    ↓
много занятых platform threads
    ↓
много памяти под стеки
    ↓
нагрузка на scheduler
    ↓
context switches
```

Теперь нужно понять, на чем физически держится WebFlux-приложение.

Когда мы пишем Spring Boot WebFlux приложение, запрос не попадает сразу в `@RestController`. До контроллера есть runtime-слой, который
принимает TCP-соединение, читает байты, превращает их в HTTP-запрос и только потом передает управление в Spring.

Упрощенно стек выглядит так:

```text
Spring Boot WebFlux
   ↓
Spring WebFlux
   ↓
Reactor Netty
   ↓
Netty
   ↓
Java NIO / native transport
   ↓
OS socket API
```

То есть центральный фундамент реактивного WebFlux-приложения — это не сам `Mono` и не сам `Flux`.

`Mono` и `Flux` — это API, через который мы описываем асинхронную работу.

А физический runtime, который принимает сетевые события, работает вокруг Netty и его EventLoop-модели.

### 9.1. Кто такой Netty в этой истории?

Netty — это сетевой framework, который позволяет строить высокопроизводительные TCP/HTTP-серверы и клиенты на event-driven non-blocking
модели.

Для нас сейчас важно не знать весь Netty API, а понять несколько базовых сущностей:

```text
EventLoopGroup
EventLoop
Channel
ChannelPipeline
Handler
```

1) Регистрация Channel в Event Loop
![cpu_threads.png](../../../../../../resources/static/lesson01/event_loop_working_2.png)

2) Как только

Их можно представить так:

```text
Netty Server
┌────────────────────────────────────────────────────────────┐
│                                                            │
│  Boss EventLoopGroup                                       │
│      принимает новые соединения                            │
│                                                            │
│  Worker EventLoopGroup                                     │
│      обрабатывает I/O уже принятых соединений              │
│                                                            │
│  Channel                                                   │
│      абстракция над конкретным соединением                 │
│                                                            │
│  ChannelPipeline                                           │
│      цепочка обработчиков вокруг Channel                   │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

Если совсем на пальцах:

```text
ServerSocket принимает соединение
        ↓
Netty создает Channel
        ↓
Channel назначается на EventLoop
        ↓
EventLoop начинает обслуживать события этого Channel
        ↓
байты проходят через ChannelPipeline
        ↓
в итоге запрос попадает в Spring WebFlux
```

### 9.2. EventLoopGroup

`EventLoopGroup` — это группа EventLoop-потоков.

В обычной серверной модели Netty можно думать так:

```text
Boss EventLoopGroup
    отвечает за accept новых TCP-соединений

Worker EventLoopGroup
    отвечает за read/write событий по уже принятым соединениям
```

Схема:

```text
Incoming connections
        ↓
┌──────────────────────┐
│ Boss EventLoopGroup  │
│ accept connections   │
└──────────────────────┘
        ↓
┌────────────────────────┐
│ Worker EventLoopGroup  │
│ read/write processing  │
└────────────────────────┘
        ↓
Channels
```

Но в Spring Boot WebFlux мы обычно не создаем Netty server руками.
Это за нас делает Reactor Netty.

Нам важно понимать, что под капотом есть небольшое количество специальных I/O-потоков.
Их задача — быстро обслуживать сетевые события, а не выполнять тяжелую бизнес-логику.

У Reactor Netty по умолчанию используется Event Loop Group, где число worker threads примерно привязано к количеству доступных процессоров,
но с минимальным значением. Поэтому в логах WebFlux-приложения можно увидеть имена вроде:

```text
reactor-http-nio-1
reactor-http-nio-2
reactor-http-nio-3
reactor-http-nio-4
```

Это не «потоки на каждый запрос». Это долгоживущие I/O-потоки.

### 9.3. EventLoop

EventLoop — это один поток, который крутится в цикле и обрабатывает события.

Упрощенный псевдокод:

```text
while (running) {
    waitForIoEvents()
    processReadyChannels()
    runTasks()
    runScheduledTasks()
}
```

Или еще проще:

```text
EventLoop:
  1. ждет сетевые события
  2. получает готовые Channel-ы
  3. читает входящие байты
  4. прогоняет байты через Pipeline
  5. вызывает нужные обработчики
  6. пишет ответ
  7. выполняет задачи из очереди
  8. снова возвращается к ожиданию событий
```

Главная идея:

```text
EventLoop — это не поток на один запрос.

EventLoop — это поток, который обслуживает много соединений.
```

То есть не так:

```text
Request-1 → Thread-1
Request-2 → Thread-2
Request-3 → Thread-3
Request-4 → Thread-4
```

А так:

```text
Channel-1 ┐
Channel-2 ├── EventLoop-1
Channel-3 ┘

Channel-4 ┐
Channel-5 ├── EventLoop-2
Channel-6 ┘
```

Один EventLoop может обслуживать много Channel-ов.
Но внутри одного EventLoop выполнение последовательное.

Это критически важно.

Если EventLoop сейчас обрабатывает Channel-1 и мы внутри этой обработки сделали `Thread.sleep(1000)`, то Channel-2 и Channel-3, назначенные
на этот же EventLoop, тоже будут ждать.

### 9.4. Channel

`Channel` — это Netty-абстракция над соединением.

Можно думать так:

```text
TCP connection
    ↓
Netty Channel
```

Когда клиент подключился к серверу, Netty создает для этого соединения Channel.

```text
Client A ───── Channel A
Client B ───── Channel B
Client C ───── Channel C
```

Дальше Channel закрепляется за EventLoop.

```text
EventLoop-1
   ├── Channel A
   ├── Channel B
   └── Channel C
```

Важное следствие:

```text
Если EventLoop заблокирован,
страдают все Channel-ы, которые он обслуживает.
```

Именно поэтому в WebFlux нельзя писать blocking-код на event-loop потоке.

### 9.5. ChannelPipeline

`ChannelPipeline` — это цепочка обработчиков вокруг Channel.

Когда в сокет прилетают байты, приложение не получает сразу готовый JSON-объект.
Сначала эти байты должны пройти через несколько уровней обработки.

Упрощенно inbound-путь выглядит так:

```text
Inbound bytes
   ↓
HTTP decoder
   ↓
HTTP request object
   ↓
Reactor Netty handler
   ↓
Spring WebFlux adapter
   ↓
WebHandler / DispatcherHandler
   ↓
@RestController
```

Outbound-путь обратно:

```text
Controller result
   ↓
Spring encoder
   ↓
Reactor Netty response writer
   ↓
HTTP encoder
   ↓
Outbound bytes
   ↓
Socket
```

Схема:

```text
Channel
┌──────────────────────────────────────────────────────────┐
│ ChannelPipeline                                          │
│                                                          │
│ inbound:                                                 │
│ bytes → decoder → http request → Spring handler          │
│                                                          │
│ outbound:                                                │
│ response object → encoder → bytes                        │
└──────────────────────────────────────────────────────────┘
```

Для первого урока нам не нужно глубоко изучать все Handler-ы.
Сейчас важно понять саму идею:

```text
ChannelPipeline — это путь превращения сетевых байтов в request
и request/response объектов обратно в байты.
```

### 9.6. Очереди EventLoop

Теперь самая важная часть.

EventLoop — это не просто:

```text
while(true) {
    selector.select()
}
```

У него есть несколько источников работы.

Упрощенно можно представить так:

```text
EventLoop
┌────────────────────────────────────────────────────────────┐
│                                                            │
│  1. I/O events                                             │
│     - Channel стал readable                                │
│     - Channel стал writable                                │
│     - пришло новое соединение                              │
│                                                            │
│  2. Task Queue                                             │
│     - обычные задачи, отправленные в EventLoop             │
│     - callback-и                                           │
│     - продолжения обработки                                │
│                                                            │
│  3. Scheduled Task Queue                                   │
│     - timeout                                              │
│     - delay                                                │
│     - задачи, которые надо выполнить позже                 │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

Важно уточнить: `I/O events` — это не совсем обычная очередь Java-объектов.
На уровне Java NIO это скорее набор готовых ключей, которые вернул Selector.
То есть Selector говорит EventLoop:

```text
Вот список Channel-ов, у которых что-то произошло.
```

А уже EventLoop по очереди их обрабатывает.

Упрощенный цикл можно представить так:

```text
while (running) {
    selectedKeys = selector.select(timeout)

    for (key in selectedKeys) {
        processIoEvent(key)
    }

    for (task in taskQueue) {
        run(task)
    }

    for (scheduledTask in expiredScheduledTasks) {
        run(scheduledTask)
    }
}
```

Слушателю нужно запомнить простую мысль:

```text
EventLoop последовательно обрабатывает:
  1. сетевые события;
  2. обычные задачи;
  3. отложенные задачи.
```

И если одна из задач блокирует EventLoop, все остальные задачи ждут.

### 9.7. Что лежит в Task Queue?

Task Queue — это очередь обычных задач, которые должны быть выполнены на этом EventLoop.

Например:

```text
- выполнить callback;
- продолжить обработку после асинхронного события;
- выполнить пользовательскую задачу, отправленную в EventLoop;
- выполнить часть pipeline-логики;
- обработать write operation;
```

Это очень важно для понимания Reactor/WebFlux.

Когда мы пишем reactive pipeline, продолжение цепочки часто будет выполнено не «где-то магически», а конкретным потоком.
Если continuation попадает в EventLoop, то оно конкурирует за время этого же EventLoop с I/O-событиями и другими задачами.

Схема:

```text
EventLoop-1

I/O event:
  Channel-1 readable
        ↓
Pipeline handler
        ↓
Spring handler
        ↓
Controller returned Mono
        ↓
later continuation scheduled
        ↓
Task Queue
        ↓
EventLoop executes continuation
```

Значит, если continuation тяжелый или blocking, он мешает не только одному запросу, а всему EventLoop.

### 9.8. Что лежит в Scheduled Task Queue?

Scheduled Task Queue — это задачи, которые должны выполниться не сейчас, а позже.

Примеры:

```text
- timeout;
- delay;
- scheduled reconnect;
- delayed write;
- periodic task;
```

Например, когда мы пишем:

```kotlin
Mono.delay(Duration.ofSeconds(1))
```

идея похожая:

```text
не держать поток одну секунду,
а запланировать событие на будущее
```

Конкретно где именно будет выполнен timer — зависит от Scheduler-а Reactor.
Но концептуально важно вот что:

```text
delay/timeout не должны означать Thread.sleep.
delay/timeout должны означать scheduled event.
```

Разница принципиальная:

```text
Thread.sleep(1000)
    поток реально спит и ничего не может делать

scheduled delay
    задача поставлена на будущее,
    поток свободен делать другую работу
```

### 9.9. Один EventLoop и пять клиентов

Представим, у нас один EventLoop и пять клиентов одновременно отправили запросы.

До запросов:

```text
EventLoop-1
   ↓
selector.select()
   ↓
ждет событий
```

Потом пришли запросы:

```text
OS/network layer:
  Channel-1 ready
  Channel-2 ready
  Channel-3 ready
  Channel-4 ready
  Channel-5 ready
```

EventLoop проснулся:

```text
EventLoop-1 loop iteration:

selected keys:
  [Channel-1, Channel-2, Channel-3, Channel-4, Channel-5]

process Channel-1
process Channel-2
process Channel-3
process Channel-4
process Channel-5

run task queue
run scheduled tasks

go back to select()
```

Внутри одного EventLoop работа идет последовательно.

```text
Channel-1 processing ──▶ Channel-2 processing ──▶ Channel-3 processing ──▶ Channel-4 processing ──▶ Channel-5 processing
```

Если обработка Channel-1 занимает 1 миллисекунду — всё хорошо.

Если обработка Channel-1 делает blocking-вызов на 1 секунду — остальные Channel-ы тоже ждут.

```text
Channel-1: Thread.sleep(1000)
              ↓
EventLoop заблокирован
              ↓
Channel-2 ждет
Channel-3 ждет
Channel-4 ждет
Channel-5 ждет
```

Вот почему главное правило WebFlux:

```text
Нельзя блокировать EventLoop.
```

### 9.10. Что значит “заблокировать EventLoop”

Плохо:

```kotlin
@GetMapping("/bad")
fun bad(): Mono<String> {
    Thread.sleep(1000)
    return Mono.just("bad")
}
```

Здесь `Thread.sleep` выполняется до возврата `Mono`.
Если controller method был вызван на event-loop потоке, мы заблокировали EventLoop.

Еще хуже, потому что выглядит реактивно:

```kotlin
@GetMapping("/also-bad")
fun alsoBad(): Mono<String> {
    return Mono.just(blockingClient.call())
}
```

Проблема в том, что `blockingClient.call()` выполняется сразу.
`Mono.just(...)` получает уже готовое значение.
То есть blocking-вызов произошел до создания `Mono`.

Еще один опасный вариант:

```kotlin
@GetMapping("/bad-map")
fun badMap(): Mono<String> {
    return Mono.just("request")
        .map {
            blockingClient.call()
        }
}
```

Здесь blocking-вызов спрятан внутри pipeline.
Если pipeline исполняется на event-loop потоке, мы снова блокируем EventLoop.

Аварийный мост к blocking-миру выглядит так:

```kotlin
@GetMapping("/legacy")
fun legacy(): Mono<String> {
    return Mono.fromCallable {
        blockingClient.call()
    }.subscribeOn(Schedulers.boundedElastic())
}
```

Но это важно правильно объяснить.

`boundedElastic` — это не “магическая реактивность”.
Это отдельный пул потоков, куда мы уносим blocking-код, чтобы не убить EventLoop.

Правильная мысль:

```text
Лучше использовать полностью non-blocking client/driver.

Если приходится вызвать legacy blocking API,
его нужно изолировать от EventLoop.
```

### 9.11. Почему EventLoop — фундамент для понимания WebFlux

Теперь можно сформулировать главное:

```text
Spring WebFlux требует другого стиля кода не потому, что авторам захотелось вернуть Mono.

Он требует другого стиля кода потому, что под ним работает event-loop runtime.

EventLoop эффективен только при одном условии:
его нельзя надолго занимать.
```

В MVC можно мыслить так:

```text
Этот request получил свой worker thread.
Если поток подождет БД — неприятно, но это проблема этого request.
```

В WebFlux нужно мыслить иначе:

```text
Этот request обрабатывается на event-loop потоке.
Если я заблокирую этот поток,
я могу задержать не только этот request,
но и другие Channel-ы, которые сидят на том же EventLoop.
```

Именно поэтому реактивный код должен быть:

```text
коротким,
неблокирующим,
композиционным,
основанным на callback/signal continuation,
а не на ожидании результата внутри потока.
```

### 9.12. Мост к следующему разделу: сетевка под капотом

Теперь у нас появляется естественный вопрос:

```text
EventLoop обрабатывает готовые Channel-ы.

Но откуда EventLoop знает, что Channel готов?
Как он понимает, что в сокет прилетели байты?
Почему ему не нужно бегать по всем соединениям вручную?
```

Ответ на этот вопрос — Selector, Socket, selected keys и I/O multiplexing.

И вот теперь мы готовы перейти к сетевке под капотом.


## 10. Сетёвка «под капотом» для разработчика: от Socket до Selector

#### 1. Что такое Сокет на самом деле?

Представьте, что **IP-адрес** — это адрес огромного многоквартирного дома. Но чтобы письмо дошло до конкретного жильца, нужен номер
квартиры.
**Порт** — это номер квартиры. **Сокет** — это «дверь» в эту квартиру.

На уровне ОС Socket — это программный интерфейс(дескриптор), через который приложение общается с сетевой картой компьютера.

Внутри сокета есть два буфера:

1) Inbound (для получения данных)
2) Outbound (для отправления данных во внешний мир).

Когда данные приходят по сети, сетевая карта и ядро ОС участвуют в доставке этих данных до сетевого стека ОС. После обработки TCP/IP
стеком данные становятся доступны приложению через receive-буфер сокета.

На пальцах можно сказать так: данные оказываются в inbound/receive-буфере сокета, откуда приложение может их прочитать.

#### 2. Проблема «Ждуна» (Blocking I/O)

В классической модели (как в старом Tomcat), когда мы говорим socket.read(), поток «засыпает» и прилипает к этой двери. Если клиент
медленный и шлёт данные по чайной ложке, поток будет висеть, пока буфер не заполнится, и именно таким образом он и ждет данные от
внешнего сервиса.

Минус: 1000 медленных клиентов = 1000 спящих потоков. А мы помним, что каждый поток — это минимум 1 МБ памяти под стек и огромные затраты на
переключение контекста (Context Switch).

#### 3. Революция: Non-blocking и Selector

В Non-blocking модели мы говорим сокету: «Если данных нет — не держи меня, я пойду проверю другие дела».

Но как понять, у какой из 1000 «дверей» (сокетов) появились данные? Не бегать же по кругу, опрашивая каждого (это съест CPU).

Для этого есть Selector. Пока, на уровне разработки, давайте понимать его вот как:

**Selector — это «вахтёр у пульта»**

Вместо того чтобы 1000 потоков стояли у 1000 дверей, мы сажаем одного «вахтёра» (Selector) перед пультом с лампочками. Каждый сокет
регистрируется у него

- Лампочка загорелась — значит, в сокет прилетели данные
- Selector говорит нашему EventLoop потоку: «Эй, 42-й сокет готов, обработай его!».

Для инженера, разбирающегося в сетях, Selector — это мультиплексор ввода-вывода. В Java это обертка над системными вызовами ядра, на
уровне разработки я бы остановился на нашем понимании на пальцах и не лез бы в сложные определения, будет полезнее, если мы поймем это
поведение на уровне железа и ОС.

Как это работает на уровне железа и ОС:

Чуть забегу вперед и сначала покажу вот такой псевдо-код

```text
while (true) {
    selector.select();
    for (readyKey : selector.selectedKeys()) {
        handle(readyKey);
    }
}
```

1) Регистрация события: Когда в Netty создается Channel (а это по сути тот же сокет, но его более высокоуровневая абстракция), он
   регистрируется в Селекторе. Мы говорим ядру: «Меня интересует событие ЧТЕНИЯ ДАННЫХ (OP_READ) (готовность к чтению) на этом дескрипторе».
2) Системный вызов select() / epoll_wait(): Поток Event Loop выполняет этот вызов. В этот момент управление передается ядру ОС. Поток
   блокируется (уходит в состояние ожидания), но не на одном сокете, а на всём наборе зарегистрированных дескрипторов.
3) Прерывание: Как только сетевая карта получает пакет, она генерирует прерывание, ОС кладет данные в буфер сокета и помечает его как
   «готовый».
4) Пробуждение: Вызов select() возвращается, поток просыпается и получает от ядра список Selected Keys — массив идентификаторов тех сокетов,
   где реально что-то произошло.

#### 4. Как Selector кормит EventLoop событиями

EventLoop мы уже разобрали выше.
Теперь посмотрим, как он получает I/O-события от ОС.

Упрощенно:

```text
socket registered in selector
        ↓
network data arrives
        ↓
OS marks socket readable
        ↓
selector.select() returns selected key
        ↓
EventLoop processes Channel
```

То есть Selector — это механизм, который помогает одному потоку не стоять у одного сокета, а ждать события сразу по множеству сокетов.

В blocking-модели поток говорит:

```text
Я буду ждать данные вот из этого конкретного socket.read()
```

В non-blocking/event-loop модели поток говорит:

```text
Я хочу знать, у каких из зарегистрированных сокетов появились события.
Когда появятся — верни мне список готовых.
```

Именно поэтому EventLoop может обслуживать много соединений небольшим количеством потоков.

#### 5. Реактивный алгоритм обращения во внешний сервис

Допустим, нам нужно в процессе обработки входящего запроса сходить во внешний микросервис.

В blocking-модели это выглядит так:

```text
request-thread
    ↓
send HTTP request to external service
    ↓
wait response
    ↓
continue
```

Поток удерживается до тех пор, пока внешний сервис не ответит.

В reactive/non-blocking модели идея другая:

```text
EventLoop initiates outbound request
        ↓
outbound Channel registered
        ↓
interest in response event is registered
        ↓
EventLoop is free
        ↓
external response arrives later
        ↓
Selector marks Channel ready
        ↓
EventLoop continues pipeline
        ↓
response to original client is completed
```

На пальцах:

```text
EventLoop не сидит и не ждет account-service.

Он говорит:
"Когда account-service ответит — продолжим вот эту цепочку".
```

И именно тут становится понятнее смысл `Mono`.

`Mono` — это не отдельный поток и не готовое значение.
`Mono` — это описание асинхронного вычисления, которое завершится позже: значением, пустым завершением или ошибкой.

#### 6. Золотое правило: не блокировать EventLoop

Ни в коем случае нельзя блокировать EventLoop.

Если внутри кода вызвать:

```kotlin
Thread.sleep(1000)
```

или вызвать blocking JDBC/HTTP client, или сделать тяжелое CPU-вычисление прямо на event-loop потоке, мы остановим один из важнейших потоков,
предназначенных для обслуживания множества соединений.

Пока EventLoop застрял на нашей blocking-операции, он не может:

- читать данные других Channel-ов;
- выполнять другие pipeline handlers;
- писать ответы;
- запускать задачи из своей очереди;
- обрабатывать scheduled tasks.

Итог:

```text
Один плохой blocking-вызов
    ↓
занятый EventLoop
    ↓
задержки у многих соединений
    ↓
рост latency
    ↓
падение throughput
```

Поэтому EventLoop должен делать короткую работу: принять событие, быстро обработать его, запланировать продолжение, передать тяжелое или
blocking-вычисление в другой пул, если другого выхода нет.

## 11. Практическая демонстрация: blocking controller vs reactive controller

Теперь после OS-модели, EventLoop и сетевки можно показать разницу руками.

Нам нужны две группы экспериментов:

```text
1. Inbound load:
   как приложение реагирует на много входящих запросов.

2. Outbound fan-out:
   как приложение ходит во множество внешних сервисов.
```

### 11.1. Inbound load: блокирующий endpoint

```kotlin
@RestController
@RequestMapping("/demo")
class BlockingController {

    @GetMapping("/blocking-sleep")
    fun blockingSleep(): String {
        log("blockingSleep: start")
        Thread.sleep(1000)
        log("blockingSleep: end")
        return "ok"
    }

    private fun log(message: String) {
        println("$message | ${Thread.currentThread().name}")
    }
}
```

Что здесь важно:

```text
Thread.sleep(1000)
    ↓
поток реально занят ожиданием
    ↓
если это event-loop thread — мы ломаем модель
```

Для учебной демонстрации этот endpoint нужен, чтобы показать сам факт удержания потока.

### 11.2. Inbound load: реактивный endpoint

```kotlin
@RestController
@RequestMapping("/demo")
class ReactiveController {

    @GetMapping("/reactive-delay")
    fun reactiveDelay(): Mono<String> {
        log("reactiveDelay: controller entered")

        return Mono.delay(Duration.ofSeconds(1))
            .doOnSubscribe { log("reactiveDelay: subscribed") }
            .doOnNext { log("reactiveDelay: delay emitted") }
            .thenReturn("ok")
            .doFinally { signal -> log("reactiveDelay: finally $signal") }
    }

    private fun log(message: String) {
        println("$message | ${Thread.currentThread().name}")
    }
}
```

Что здесь важно:

```text
Mono.delay(Duration.ofSeconds(1))
    ↓
не означает Thread.sleep
    ↓
задача планируется на будущее
    ↓
поток не должен стоять одну секунду
```

Команды для нагрузки:

```bash
hey -n 1000 -c 100 http://localhost:8080/demo/blocking-sleep
```

```bash
hey -n 1000 -c 100 http://localhost:8080/demo/reactive-delay
```

Что смотрим:

```text
- requests/sec
- average latency
- p95/p99 latency
- active threads
- CPU usage
- memory
- thread names in logs
```

Важно честно проговорить:

```text
Thread.sleep vs Mono.delay — это не честный бизнес-бенчмарк.

Это учебная демонстрация:
Thread.sleep удерживает поток,
Mono.delay планирует событие и освобождает поток.
```

### 11.3. Outbound fan-out: реальный микросервисный сценарий

Теперь более честный пример.

Есть endpoint:

```text
GET /profile/{id}
```

Чтобы собрать profile, сервис должен сходить в несколько внешних сервисов:

```text
client-service
account-service
limit-service
transaction-service
notification-service
```

#### Blocking sequential-вариант

```kotlin
@GetMapping("/blocking-profile/{id}")
fun blockingProfile(@PathVariable id: String): ProfileResponse {
    val client = blockingClientService.getClient(id)
    val account = blockingAccountService.getAccount(id)
    val limit = blockingLimitService.getLimit(id)
    val transactions = blockingTransactionService.getTransactions(id)
    val notifications = blockingNotificationService.getNotifications(id)

    return ProfileResponse.of(client, account, limit, transactions, notifications)
}
```

Если каждый внешний сервис отвечает примерно 300 ms, последовательный blocking-вариант даст примерно:

```text
300 ms * 5 = около 1500 ms
```

Это не потому, что Java медленная.
Это потому, что мы сами заставили запрос ждать внешние сервисы по очереди.

#### Blocking parallel-вариант

Можно ускорить blocking-подход через `ExecutorService` или `CompletableFuture`.

```text
5 external calls
    ↓
5 blocking tasks
    ↓
5 worker threads wait responses
```

Latency может стать лучше, но цена другая:

```text
100 concurrent incoming requests
5 outbound blocking calls на request
    ↓
до 500 blocking tasks
    ↓
много потоков или большая очередь задач
```

То есть мы уменьшили latency одного запроса, но увеличили давление на thread pools.

#### Reactive fan-out через WebClient

```kotlin
@GetMapping("/reactive-profile/{id}")
fun reactiveProfile(@PathVariable id: String): Mono<ProfileResponse> {
    val clientMono = clientWebClient.getClient(id)
    val accountMono = accountWebClient.getAccount(id)
    val limitMono = limitWebClient.getLimit(id)
    val transactionsMono = transactionWebClient.getTransactions(id)
    val notificationsMono = notificationWebClient.getNotifications(id)

    return Mono.zip(
        clientMono,
        accountMono,
        limitMono,
        transactionsMono,
        notificationsMono
    ).map { tuple ->
        ProfileResponse.of(
            tuple.t1,
            tuple.t2,
            tuple.t3,
            tuple.t4,
            tuple.t5
        )
    }
}
```

Если каждый внешний сервис отвечает примерно 300 ms, reactive fan-out теоретически должен быть ближе к:

```text
max(300 ms, 300 ms, 300 ms, 300 ms, 300 ms) = около 300 ms
```

Но важно объяснить:

```text
Это не потому, что WebFlux ускорил сеть.

Это потому, что мы запустили независимые I/O операции конкурентно
и не удерживаем отдельный поток на каждое ожидание.
```

### 11.4. Что сравниваем цифрами

Не обещаем заранее конкретные результаты. Замеры зависят от машины, ОС, JVM, числа ядер, задержки мок-сервисов, настроек пулов и лимитов.

Но сравнивать нужно такие метрики:

```text
1. Throughput:
   сколько request/sec выдерживает приложение

2. Latency:
   avg, p50, p95, p99

3. Threads:
   сколько потоков реально активно

4. CPU:
   общий %, context switching, saturation

5. Memory:
   растет ли потребление при росте concurrency

6. Error rate:
   сколько timeout/connection refused/5xx

7. EventLoop health:
   появляются ли задержки при блокировке event-loop
```

Таблица ожидаемых выводов:

| Сценарий | Что ожидаем увидеть | Главный вывод |
|---|---|---|
| Blocking sequential outbound | высокая latency | запросы ждут внешние сервисы по очереди |
| Blocking parallel outbound | latency ниже, но потоков больше | быстрее по latency, дороже по потокам |
| Reactive WebClient + Mono.zip | latency ниже без взрыва потоков | конкурентный non-blocking I/O |
| Reactive + blocking внутри map | деградация всего приложения | event-loop нельзя блокировать |

Главная формула демонстрации:

```text
Reactive не уменьшает latency внешнего сервиса.
Reactive уменьшает стоимость ожидания этой latency внутри нашего приложения.
```
