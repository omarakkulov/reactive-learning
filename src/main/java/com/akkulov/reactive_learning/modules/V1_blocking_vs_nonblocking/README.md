# День 1. Blocking vs Non-Blocking

## Зачем этот блок

В реактивном программировании легко слишком рано начать с `Mono`, `Flux`, `map` и `flatMap`.
Но сначала важнее собрать простую внутреннюю модель: что именно мы пытаемся оптимизировать и
почему обычная модель "один запрос - один поток" начинает плохо работать при большом количестве I/O.

Цель блока: понять, почему поток не должен стоять и ждать там, где работа на самом деле происходит
не в CPU, а во внешнем мире: в сети, базе данных, файловой системе или другом сервисе.

## Бытовая модель

Представим сотрудника на ресепшене.

Blocking-подход:

```text
1. Клиент просит документ.
2. Сотрудник отправляет запрос в архив.
3. Сотрудник стоит у окна и ничего больше не делает.
4. Архив отвечает через 1 секунду.
5. Сотрудник отдает документ клиенту.
```

Non-blocking-подход:

```text
1. Клиент просит документ.
2. Сотрудник отправляет запрос в архив.
3. Сотрудник не стоит у окна, а берет следующего клиента.
4. Когда архив ответил, сотрудник возвращается к первому запросу.
5. Ответ отдается клиенту.
```

В обоих случаях архив может отвечать одну и ту же секунду. Разница не в том, что архив стал быстрее.
Разница в том, что сотрудник перестал простаивать.

## От железа к backend-коду

У приложения есть несколько разных типов работы.

CPU-bound работа:

- посчитать хэш;
- распарсить большой JSON;
- сжать данные;
- выполнить сложный алгоритм.

I/O-bound работа:

- сходить в базу данных;
- отправить HTTP-запрос;
- прочитать файл;
- дождаться ответа от брокера сообщений.

CPU работает очень быстро. RAM тоже быстрая относительно сети и диска. А вот I/O часто означает
"мы отправили запрос и ждем внешний мир". Для backend-сервиса это принципиально: большинство времени
обычный request-response код часто не считает, а ждет.

## Что такое blocking I/O

Blocking I/O означает: поток начал операцию и не может продолжать полезную работу, пока операция не
закончится.

Пример:

```java
Thread.sleep(1000);
```

Это не реальный сетевой вызов, но для учебной демонстрации он хорош: поток одну секунду ничего не
делает. Он занят формально, но не выполняет полезную CPU-работу.

В традиционной servlet-модели, например в Spring MVC, обычно удобно думать так:

```text
request -> worker thread -> controller -> service -> repository/client -> response
```

Если внутри `repository/client` происходит blocking I/O, worker thread стоит и ждет. При небольшой
нагрузке это нормально. При большой нагрузке нужно больше потоков, больше памяти под стеки, больше
переключений контекста и больше давления на планировщик ОС.

## Что такое non-blocking I/O

Non-blocking I/O означает: поток не обязан ждать завершения операции. Он может зарегистрировать интерес
к результату и вернуться к обработке других событий.

В Reactor это хорошо видно на примере:

```java
Mono.delay(Duration.ofSeconds(1))
```

Такой код не усыпляет текущий поток на одну секунду. Он создает асинхронное событие: "через секунду
продолжить pipeline". Текущий поток может освободиться.

Важно: задержка все еще длится одну секунду. Реактивность не отменяет время ожидания. Она меняет то,
как приложение распоряжается потоками во время ожидания.

## Concurrency и parallelism

Concurrency - это способность вести несколько задач в один период времени.

Parallelism - это реальное одновременное выполнение нескольких задач на разных CPU cores.

Пример с одним сотрудником:

- concurrency: сотрудник ведет несколько заявок, переключаясь между ними;
- parallelism: несколько сотрудников реально работают одновременно.

Reactive WebFlux в первую очередь помогает с concurrency для I/O-bound задач. Для CPU-bound задач
нужно думать отдельно: если работа реально грузит CPU, ее нельзя просто спрятать в реактивную цепочку
и ожидать чуда.

## Latency и throughput

Latency - сколько времени занимает один запрос.

Throughput - сколько запросов система может обработать за единицу времени.

Если внешний сервис отвечает за 300 ms, реактивный код не сделает этот ответ равным 3 ms. Но он может
помочь тому же количеству потоков обслуживать больше одновременных запросов, потому что потоки не
будут простаивать в ожидании I/O.

## Почему это важно для WebFlux

Spring WebFlux поверх Reactor Netty использует небольшое количество event loop потоков. Эти потоки
должны быстро принимать события, чуть-чуть обрабатывать их и передавать работу дальше.

Если внутри такого потока сделать blocking-вызов, например `Thread.sleep`, blocking JDBC или обычный
blocking HTTP client, мы блокируем не "один безобидный запрос", а поток, который мог обслуживать
другие соединения.

Поэтому первое правило:

```text
Не блокировать event loop.
```

Пока это правило можно понимать на уровне интуиции. Во втором блоке мы разберем путь запроса через
WebFlux, Reactor Netty, Channel и EventLoop подробнее.

## Главный вывод

Реактивность нужна не для того, чтобы любой код стал быстрее. Она нужна, чтобы приложение лучше
использовало ограниченные ресурсы, особенно потоки, когда основное время уходит на ожидание I/O.

Если задача I/O-bound, non-blocking модель помогает держать много одновременных запросов меньшим
числом потоков. Если задача CPU-bound, нужно думать про parallelism, scheduler'ы и границы выполнения.
