# МЕТОДИЧЕСКИЙ МАТЕРИАЛ ДЛЯ ЛЕКЦИИ
# Тема: Прием аргументов в @RestController Spring WebFlux (Обычный объект vs Mono)

При разработке реактивного контроллера на Spring WebFlux программист может написать два варианта метода для обработки POST/PUT запросов с телом (JSON). Внешне они похожи, но под капотом ведут себя принципиально по-разному с точки зрения работы с байтами и потоками Event Loop.

---

## ВАРИАНТ 1: Принимаем готовую модельку (Императивный стиль в реактивном фреймворке)

@PostMapping("/animal")
public Mono<String> createAnimal(@RequestBody Animal animal) {
return Mono.just("Создано животное с возрастом: " + animal.getAge());
}

### Что происходит за кулисами "на пальцах":
1. Сетевая карта принимает HTTP-заголовки. Netty понимает, что это POST-запрос, но самого тела (JSON) в памяти еще нет — оно все еще летит по сети от клиента мелкими TCP-пакетами.
2. Spring WebFlux НЕ ВЫЗЫВАЕТ ваш метод контроллера сразу.
3. Фреймворк встает в режим ожидания. Он ждет, пока Reactor Netty соберет абсолютно ВСЕ входящие байты тела запроса из сетевого сокета.
4. Когда все байты долетели, активируется кодек (например, библиотека Jackson). Он берет этот массив байтов, парсит JSON и конструирует в памяти Java-объект new Animal().
5. Только теперь Spring передает этот готовый объект в ваш метод createAnimal(Animal animal).

Итог: Метод контроллера вызывается только тогда, когда данные ПОЛНОСТЬЮ физически находятся в памяти сервера в виде Java-объекта.

---

## ВАРИАНТ 2: Принимаем реактивный Mono (Истинный реактивный стиль)

@PostMapping("/animal-reactive")
public Mono<String> createAnimalReactive(@RequestBody Mono<Animal> animalMono) {
return animalMono.map(animal -> "Создано животное с возрастом: " + animal.getAge());
}

### Что происходит за кулисами "на пальцах":
1. Сетевая карта принимает ТОЛЬКО HTTP-заголовки. Тело запроса (JSON) еще даже не начинало скачиваться или находится в процессе передачи.
2. Spring WebFlux МГНОВЕННО вызывает ваш метод контроллера. Он не ждет байтов из сети!
3. Вместо готового объекта Spring протягивает в метод пустую "трубу" или "обещание" — объект Mono<Animal>. Внутри этого Mono еще ничего нет.
4. Ваш метод за доли микросекунды выполняет свой код: он просто прикрепляет к этой трубе инструкцию (через .map(), .flatMap()), говоря: "Когда сюда наконец прилетит животное, достань его возраст и сделай строку".
5. Метод завершается и возвращает результирующий Mono обратно фреймворку. Поток EventLoop мгновенно освобождается и идет обслуживать других пользователей. Он не ждал сеть!
6. Что происходит позже: Когда клиент по кабелю наконец досылает байты JSON, Netty их ловит, Jackson на лету конвертирует их в объект Animal и этот объект "вбрасывается" в ту самую трубу (Mono). Инструкция .map(), которую вы написали заранее, автоматически срабатывает, и ответ улетает клиенту.

Итог: Метод контроллера вызывается ДО того, как тело запроса физически скачалось из сети. Мы работаем не с данными, а с декларативным рецептом их обработки.

---

## СРАВНЕНИЕ ДЛЯ СТУДЕНТОВ: Зачем усложнять и принимать Mono?

* Защита от медленных клиентов: Если пользователь загружает большой JSON через плохой мобильный интернет (3G/Edge), в Вариание 1 сервер будет дольше удерживать ресурсы в ожидании сборки объекта. В Варианте 2 поток Event Loop вообще не замечает задержек сети — он просто раскидал "обещания" и свободен.
* Эффективный пайплайн: Принятие Mono позволяет бесшовно связать сетевой поток ввода от клиента напрямую с потоком вывода (например, перенаправить этот Mono дальше во внешнюю БД или другой микросервис через WebClient), вообще не материализуя данные в памяти целиком в один конкретный момент времени.
