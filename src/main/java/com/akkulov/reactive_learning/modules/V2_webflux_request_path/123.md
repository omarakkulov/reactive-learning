# Урок 2. WebFlux Request Path

Главная задача второго урока - дополнить первый урок конкретным путем HTTP-запроса в Spring WebFlux.

На первом уроке мы разобрали, почему blocking I/O плохо использует потоки и почему EventLoop нельзя блокировать.
Теперь нужно ответить на следующий вопрос:

```text
Если EventLoop не должен ждать запрос от начала до конца,
то как WebFlux вообще доводит HTTP request до HTTP response?
```

К концу урока человек должен уметь объяснить WebFlux request path так же уверенно, как в Spring MVC обычно объясняют путь через
`DispatcherServlet`.

## 0. Что человек должен понять после урока

После этого урока слушатель должен уметь объяснить:

- что такое Netty server в WebFlux-приложении;
- зачем нужны `EventLoopGroup` и `EventLoop`;
- что такое `Channel`;
- что делает `ChannelPipeline`;
- зачем существует Reactor Netty;
- где начинается Spring WebFlux поверх Reactor Netty;
- какую роль играет `HttpHandler`;
- что такое `WebHandler`;
- зачем нужны `HandlerMapping` и `HandlerAdapter`;
- почему controller возвращает `Mono` или `Flux`;
- кто подписывается на publisher и как сигналы превращаются в HTTP response.

Главная формула урока:

```text
WebFlux request не "сидит" на одном потоке.
Он проходит через Channel, события, handlers, reactive pipeline и publisher signals.
```

## 1. Возвращаемся к примеру `/profile/{id}`

Начать второй урок лучше не с терминов, а с вопроса из предыдущей лекции.

Допустим, у нас есть endpoint:

```kotlin
@GetMapping("/profile/{id}")
fun profile(@PathVariable id: String): Mono<ProfileResponse> {
    return Mono.just(id)
        .map { validate(it) }
        .flatMap { validId ->
            externalClient.getProfile(validId)
        }
        .map { externalResponse ->
            ProfileResponse.from(externalResponse)
        }
}
```

В MVC мы часто думаем так:

```text
request живет на worker thread
```

То есть worker thread вошел в controller, сходил в service, repository/client, дождался результата и вернул response.

В WebFlux мы не хотим, чтобы EventLoop ждал внешний сервис.

Значит, нужно задать вопрос:

```text
Если поток не ждет внешний сервис,
где находится состояние запроса?
Кто помнит, что Client-1 еще нужно отдать response?
```

Ответ, как mental model:

```text
Состояние запроса живет не в заблокированном стеке EventLoop-потока,
а в framework objects, Channel state, reactive pipeline, subscription и callback'ах.
```

Это не математически полное определение, но для backend-разработчика это очень полезная модель.

## 2. Последовательный EventLoop не означает последовательное завершение запросов

В первом уроке мы говорили:

```text
Один EventLoop может обслуживать много Channel-ов.
Внутри одного EventLoop выполнение последовательное.
```

Теперь уточняем:

```text
последовательно выполняются отдельные события и callback'и,
а не весь HTTP-запрос от входа до ответа.
```

Если пришли:

```text
GET /profile/1
GET /profile/2
```

то старт обработки может быть последовательным:

```text
t1:
  Channel-A ready
  validate(1)
  start outbound call X
  return to EventLoop

t2:
  Channel-B ready
  validate(2)
  start outbound call Y
  return to EventLoop
```

Но завершение может быть в другом порядке:

```text
t3:
  outbound Channel-Y ready first
  continue /profile/2
  write response to Client-2

t4:
  outbound Channel-X ready later
  continue /profile/1
  write response to Client-1
```

Формула:

```text
Sequential callback execution != ordered request completion
```

Или:

```text
EventLoop sequential по выполнению кода.
WebFlux concurrent по состоянию многих запросов.
```

## 3. Аккуратно про "очередь EventLoop"

Вопрос, который естественно возникает после первого урока:

```text
Входящие и исходящие запросы падают в очередь EventLoop как events?
```

Ответ:

```text
Почти, но важно не смешать уровни.
```

Не очень точная формулировка:

```text
HTTP-запросы складываются в очередь EventLoop.
```

Более точная формулировка:

```text
Channel зарегистрирован на EventLoop.
Когда Channel готов к чтению или записи, EventLoop получает I/O readiness event
и последовательно запускает нужные handlers/callbacks.
```

У EventLoop есть два основных источника работы:

```text
1. I/O readiness events от Selector:
   - new connection accepted;
   - Channel ready for read;
   - Channel ready for write.

2. Task queue:
   - callbacks;
   - scheduled tasks;
   - tasks, отправленные через eventLoop.execute(...);
   - внутренние задачи Netty/Reactor.
```

HTTP request в привычном Spring-смысле появляется выше:

```text
bytes пришли в socket buffer
  -> Selector сообщил: Channel ready for read
  -> EventLoop прочитал bytes из Channel
  -> ChannelPipeline декодировал bytes как HTTP
  -> Reactor Netty поднял это до HTTP-level request
  -> Spring WebFlux получил request
```

Поэтому для лекции лучше говорить:

```text
EventLoop исполняет не "запросы целиком", а куски работы по событиям.
```

## 4. Общая карта request path

Весь путь можно держать в голове как одну большую схему:

```text
Client
 -> Socket
 -> Netty Channel
 -> EventLoopGroup
 -> EventLoop
 -> ChannelPipeline
 -> Reactor Netty HttpServer
 -> Spring HttpHandler
 -> WebHandler
 -> HandlerMapping
 -> HandlerAdapter
 -> Controller method
 -> Mono/Flux
 -> WebFlux infrastructure subscribes
 -> onNext/onError/onComplete
 -> encode response body
 -> Netty Channel write
 -> Client
```

Сначала мы разберем нижнюю часть: Netty.

Потом среднюю часть: Reactor Netty.

Потом верхнюю часть: Spring WebFlux.

И в конце вернемся к вопросу:

```text
кто подписывается на Mono/Flux и как response уходит клиенту?
```

## 5. Netty server

В WebFlux-приложении, если мы используем стандартный reactive stack, HTTP-сервером обычно является Reactor Netty.

Под ним работает Netty.

Netty - это network framework, который умеет строить event-driven non-blocking серверы и клиенты.

На уровне идеи Netty server делает несколько вещей:

- открывает server socket;
- принимает новые TCP-соединения;
- создает `Channel` для каждого соединения;
- регистрирует `Channel` на одном из `EventLoop`;
- прогоняет входящие и исходящие события через `ChannelPipeline`.

Важно:

```text
Netty пока ничего не знает о наших `@GetMapping`.
Netty работает на уровне socket, Channel, bytes, events и handlers.
```

## 6. EventLoopGroup и EventLoop

`EventLoopGroup` - это группа EventLoop-потоков.

Упрощенно:

```text
EventLoopGroup
   ├── EventLoop-1
   ├── EventLoop-2
   ├── EventLoop-3
   └── EventLoop-4
```

Один `EventLoop` может обслуживать много `Channel`-ов:

```text
EventLoop-1
   ├── Channel-A
   ├── Channel-B
   └── Channel-C
```

Внутри одного EventLoop выполнение кода последовательное:

```text
one EventLoop thread -> one callback at a time
```

Но много Channel-ов могут находиться в прогрессе одновременно с точки зрения I/O ожидания:

```text
Channel-A ждет внешний сервис
Channel-B ждет request body
Channel-C готов к записи response
```

EventLoop не обязан ждать ни один из них. Он возвращается к циклу и берет следующую готовую работу.

## 7. Channel

`Channel` - это Netty-абстракция над соединением.

Можно думать так:

```text
TCP connection
  -> Netty Channel
```

Для inbound request:

```text
Client -> inbound Channel -> MyApp
```

Для outbound WebClient call:

```text
MyApp -> outbound Channel -> external-service
```

В примере `/profile/{id}`:

```text
/profile/1:
  inbound Channel-A  - соединение Client-1 с нашим приложением
  outbound Channel-X - соединение нашего приложения с external-service

/profile/2:
  inbound Channel-B  - соединение Client-2 с нашим приложением
  outbound Channel-Y - соединение нашего приложения с external-service
```

Каждый Channel зарегистрирован на EventLoop.

Когда по Channel можно читать или писать, EventLoop получает событие готовности и запускает обработку.

## 8. ChannelPipeline

`ChannelPipeline` - это цепочка обработчиков, через которые проходят события и данные Channel.

Упрощенно:

```text
bytes from socket
 -> handler-1
 -> handler-2
 -> HTTP decoder
 -> HTTP object
 -> Reactor Netty handler
```

Входящие данные идут по inbound side pipeline.

Исходящие данные идут по outbound side pipeline.

Для HTTP-сервера pipeline делает очень важную работу:

- читает bytes;
- декодирует bytes в HTTP request;
- кодирует HTTP response обратно в bytes;
- вызывает нужные handlers;
- передает события выше.

Важная мысль:

```text
Controller еще не вызывается на уровне ChannelPipeline напрямую.
ChannelPipeline поднимает сетевое событие до уровня, где Reactor Netty может передать его Spring WebFlux.
```

## 9. Reactor Netty как мост

Reactor Netty - это слой, который соединяет Netty event-driven runtime с Reactor API.

Netty работает с:

```text
Channel
ChannelPipeline
ChannelHandler
ByteBuf
events
```

Spring WebFlux хочет работать с более высокоуровневой HTTP/reactive моделью.

Reactor Netty делает мост:

```text
Netty HTTP event
 -> Reactor Netty HttpServer
 -> reactive HTTP request/response abstraction
 -> Spring WebFlux HttpHandler
```

В этот момент мы переходим от "байты пришли в Channel" к "у нас есть HTTP request, который может быть обработан reactive pipeline".

## 10. Spring WebFlux поверх Reactor Netty

Spring WebFlux не заменяет Netty.

Spring WebFlux живет выше:

```text
Netty
  -> Reactor Netty
    -> Spring WebFlux
```

Reactor Netty принимает сетевую часть и передает управление в Spring WebFlux через адаптеры.

В Spring WebFlux появляется web-level модель:

- request;
- response;
- filters;
- handler mappings;
- handler adapters;
- controller method invocation;
- codecs;
- message writers.

Теперь мы уже близко к привычному Spring-слою.

## 11. HttpHandler

`HttpHandler` - это низкоуровневый контракт Spring для обработки HTTP request/response в reactive stack.

Упрощенно:

```text
HttpHandler:
  принимает ServerHttpRequest + ServerHttpResponse
  возвращает Mono<Void>
```

Можно думать так:

```text
HttpHandler - это входная точка Spring WebFlux на HTTP-уровне.
```

Reactor Netty получает HTTP-событие от Netty и через адаптер передает его в Spring `HttpHandler`.

Важно:

```text
HttpHandler еще не означает "controller".
Это общий HTTP-level вход в WebFlux-инфраструктуру.
```

## 12. WebHandler

`WebHandler` - более высокий web-level слой.

Если `HttpHandler` - вход с HTTP request/response, то `WebHandler` - уже обработка в терминах WebFlux.

Здесь появляются:

- web filters;
- exception handling;
- handler mappings;
- handler adapters;
- result handling.

Упрощенная схема:

```text
HttpHandler
  -> WebHandler
  -> WebFilter chain
  -> handler resolution
  -> handler invocation
  -> result handling
```

## 13. HandlerMapping

`HandlerMapping` отвечает на вопрос:

```text
Кто должен обработать этот request?
```

Для annotated controllers он находит метод вроде:

```kotlin
@GetMapping("/profile/{id}")
fun profile(...)
```

То есть request:

```text
GET /profile/1
```

матчится на handler:

```text
profile(id = "1")
```

В Spring MVC эту идею тоже можно встретить. Отличие WebFlux не в том, что handler mapping исчезает, а в том, что результат handler'а
дальше обрабатывается reactive способом.

## 14. HandlerAdapter

`HandlerAdapter` отвечает на вопрос:

```text
Как вызвать найденный handler?
```

`HandlerMapping` нашел, что обработчиком является controller method.

`HandlerAdapter` умеет:

- подготовить аргументы метода;
- достать `@PathVariable`;
- обработать `@RequestBody`;
- вызвать controller method;
- получить результат вызова.

В нашем примере:

```kotlin
profile(id = "1")
```

возвращает:

```text
Mono<ProfileResponse>
```

## 15. Controller возвращает Mono/Flux

В WebFlux controller обычно не возвращает готовые данные.

Он возвращает `Publisher`, чаще всего:

```text
Mono<T>  - 0 или 1 элемент
Flux<T>  - 0..N элементов
```

В нашем примере:

```kotlin
fun profile(...): Mono<ProfileResponse>
```

Это значит:

```text
controller вернул описание асинхронной работы,
которая когда-нибудь даст ProfileResponse или ошибку.
```

Важно:

```text
Controller не должен сам вызывать subscribe().
```

Если мы вызываем `subscribe()` руками в controller/service, мы часто отрываем выполнение от WebFlux-инфраструктуры: ломаем управление
ошибками, cancellation, response handling и общий lifecycle запроса.

## 16. Где HTTP response подписывается на publisher

В WebFlux на `Mono/Flux`, который вернул controller, подписывается инфраструктура.

Упрощенно:

```text
Controller returns Mono<ProfileResponse>
  -> WebFlux infrastructure subscribes
  -> Publisher emits signals
  -> WebFlux turns signals into HTTP response
```

Сигналы:

```text
onNext(ProfileResponse)
  -> encode response body
  -> write bytes to Netty Channel

onError(error)
  -> error handling
  -> write error response

onComplete()
  -> complete HTTP response
```

Для тела ответа используются codecs / message writers.

Например:

```text
ProfileResponse
  -> JSON encoder
  -> bytes
  -> Reactor Netty response
  -> Netty Channel write
  -> socket
  -> Client
```

## 17. Outbound WebClient path в примере `/profile/{id}`

Теперь вернемся к строке:

```kotlin
externalClient.getProfile(validId)
```

Если это WebClient/Reactor Netty, то путь примерно такой:

```text
WebClient starts external call
  -> Reactor Netty берет или создает outbound Channel
  -> outbound Channel зарегистрирован на EventLoop
  -> EventLoop делает write/flush HTTP request во внешний сервис
  -> EventLoop не ждет response
  -> внешний сервис отвечает
  -> OS кладет bytes в socket receive buffer
  -> Selector сообщает: outbound Channel ready for read
  -> EventLoop читает response bytes
  -> ChannelPipeline декодирует response
  -> Mono от externalClient получает сигнал
  -> pipeline продолжается
```

После этого выполняется:

```kotlin
.map { externalResponse ->
    ProfileResponse.from(externalResponse)
}
```

А затем WebFlux пишет response клиенту через inbound Channel.

## 18. Полный путь `/profile/{id}` на одном дыхании

Теперь можно собрать все в одну историю.

```text
Client sends GET /profile/1
  -> bytes приходят в socket buffer
  -> Selector видит Channel ready for read
  -> EventLoop читает bytes из inbound Channel
  -> ChannelPipeline декодирует HTTP request
  -> Reactor Netty получает HTTP request
  -> Spring HttpHandler принимает request/response
  -> WebHandler запускает web-level обработку
  -> HandlerMapping находит controller method
  -> HandlerAdapter вызывает profile(id = "1")
  -> controller возвращает Mono<ProfileResponse>
  -> WebFlux подписывается на Mono
  -> externalClient стартует outbound non-blocking call
  -> EventLoop освобождается
  -> позже outbound Channel получает response
  -> Mono получает signal
  -> pipeline создает ProfileResponse
  -> WebFlux кодирует body
  -> Reactor Netty пишет bytes в inbound Channel
  -> Client получает HTTP response
```

Это и есть WebFlux request path.

## 19. Сравнение с Spring MVC

Spring MVC mental model:

```text
Client
  -> Servlet container
  -> worker thread
  -> DispatcherServlet
  -> HandlerMapping
  -> HandlerAdapter
  -> Controller
  -> blocking service/repository/client
  -> response
```

Условно:

```text
request часто живет на worker thread stack
```

WebFlux mental model:

```text
Client
  -> Netty Channel
  -> EventLoop
  -> ChannelPipeline
  -> Reactor Netty
  -> HttpHandler
  -> WebHandler
  -> HandlerMapping
  -> HandlerAdapter
  -> Controller
  -> Mono/Flux
  -> signals
  -> response write
```

Условно:

```text
request продвигается через события, framework state и reactive signals
```

## 20. Главный вывод

После этого урока нужно оставить слушателю несколько точных фраз.

Первая:

```text
EventLoop не владеет HTTP-запросом от начала до конца.
Он выполняет короткие куски работы по событиям.
```

Вторая:

```text
Channel - это соединение в модели Netty.
ChannelPipeline - цепочка обработчиков событий и bytes.
```

Третья:

```text
Reactor Netty соединяет Netty runtime с Reactor-style HTTP API.
```

Четвертая:

```text
Spring WebFlux начинается выше: HttpHandler, WebHandler, HandlerMapping, HandlerAdapter, Controller.
```

Пятая:

```text
Controller возвращает Mono/Flux, а WebFlux-инфраструктура подписывается на publisher
и превращает сигналы в HTTP response.
```

Финальная формула:

```text
WebFlux request path - это путь от socket events к reactive signals и обратно к HTTP response.
```
